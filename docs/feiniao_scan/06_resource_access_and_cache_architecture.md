# 飞鸟资源接入与缓存架构（收口版）

> 目标：围绕“长尾第一版 + 测试站最小链路”做架构收口，不进入业务大改。

## 1. 资源接入总体原则

- 飞鸟定位为组织层（路由、页面、长尾、SEO、模板），不作为重内容仓库。
- 数据库 / TXT / IMG 均需支持三种模式：本地模式、域名模式、跨服务器模式。
- 程序内部资源读取优先级：
  1. 本地路径 / 本地数据库
  2. 私网链路（含 Tailscale）
  3. 公网域名（兜底）
- 浏览器外部访问可继续走域名 + CF/CDN，不与程序内部读取优先级冲突。

## 2. 数据库接入规划

- 支持形态：
  - 本地库（同机）
  - 私网库（同内网 / Tailscale）
  - 远程库（跨服务器）
- 连接策略：
  - 推荐只读账号用于内容读取
  - 明确连接超时、读超时、重试上限
  - 查询遵循“最小查询原则”（按需字段、按需范围）
- 跨服务器阶段优先私网路径（含 Tailscale），不优先公网直连 DB。

## 3. TXT 接入规划

- 支持模式：
  - `local`：本地文件系统直读
  - `domain`：通过 TXT 域名对外访问
  - `remote_mount`：远端目录挂载到本地后读取
- 程序内部正文读取优先本地路径（含本地挂载路径）。
- 浏览器外部可走 TXT 域名输出。
- 不建议程序内部为读正文绕行 TXT 公网域名。

## 4. IMG 接入规划

- 支持本地原图存储 + 公共图片域名输出。
- 前台优先输出图片域名，便于统一 CDN/CF 加速。
- 服务端做文件存在判断时优先本地/内网路径。
- 图片访问链路适合长期保持“域名 + CF/CDN”模式。

## 5. 缓存架构规划

- Redis（核心对象缓存）：
  - 配置对象
  - route 结果
  - pages / longtail / keywords 相关对象
  - book_base / chapter_list
  - 锁与状态位
- Memcached（短 TTL 微缓存）：
  - 热点详情页片段
  - 热点目录页片段
  - 首页热点模块
  - 热点查询结果
- 不建议缓存项：
  - 全量章节正文
  - 全量长尾 HTML
  - 全站预热产物
- 原则：Redis 与 Memcached 可协同，但同一类核心对象不重复缓存到两边，避免失效混乱。

## 6. 垃圾缓存控制规则

- 不预生成全量缓存。
- 优先对象缓存，不优先整页缓存。
- 长尾冷页不做长 TTL HTML 缓存。
- 负缓存（空结果/不存在）TTL 必须短。
- 更新以对象级失效为主，不做全站 flush。
- key 强制带前缀：`系统:环境:站点:模块:对象`。
- 三系统同服（船说/YGBOOK/飞鸟）时，至少按“系统 + 环境 + 站点”做命名隔离。

## 7. 三系统同服承载建议

- 船说 / YGBOOK / 飞鸟 同服时：
  - Redis 负责核心对象、状态、路由关键数据。
  - Memcached 负责短 TTL 热点响应加速。
- Redis 分前缀/分 DB 仅为逻辑隔离，不等同于物理隔离。
- 流量上来后可再考虑 Redis 拆实例（按系统或按职责拆分）。
- 当前阶段先完成架构预留，不进入大规模部署改造。

## 8. 测试站最小验证建议

建议先跑通最小闭环样本：
- 1 本书
- 1 个目录
- 1 章正文
- 1 张封面
- 1 个长尾页入口

观测指标：
- MySQL 查询耗时
- Redis 命中率与耗时
- Memcached 命中率与耗时
- PHP-FPM 请求耗时
- TXT 文件读取耗时
